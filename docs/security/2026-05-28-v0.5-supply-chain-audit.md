# Security Audit — GitExpose v0.5 Supply-Chain / OSV Egress Surface

**Generated:** 2026-05-28
**Scope:** the new v0.5 outbound-egress + concurrency code — `gitexpose/supply_chain/osv.py`,
`gitexpose/verification/engine.py`, `gitexpose/verification/helpers.py`,
`gitexpose/supply_chain/lockfiles/*`, `gitexpose/supply_chain/cvss.py`, and the
`supply-chain` CLI wiring.
**Method:** `/arch → /attack → /audit` loop. Adversarial pass (vulnhunter / variant-analysis /
sharp-edges / insecure-defaults lenses), then per-candidate `fp-check`-style 5-phase verification
(data-flow → exploitability → impact → PoC → devil's advocate) plus the 6-gate review. Each
true positive carries a regression test that serves as its PoC / negative-PoC.
**Candidates reviewed:** 5  ·  **True positives:** 5  ·  **False positives:** 0
**Status:** all findings **remediated in v0.5.1** (`20dd22e`); 329 tests, CI green on Python 3.9–3.12.

## Executive summary

A focused audit of the v0.5 supply-chain intelligence feature — the first GitExpose code to make
*unsolicited* outbound requests on the user's behalf (OSV.dev). No critical or high issues were
found. The headline egress/SSRF risk is **not present**: all OSV endpoint URLs are hardcoded
module constants, no lock-file-derived data reaches URL construction, only package names+versions
(never secrets) leave the host, and the HTTP client does not follow redirects. The five issues
found are one medium-severity availability/robustness defect (F-001 — a malformed upstream response
could crash the scan instead of degrading) and four low-severity hardening items (input encoding,
two resource-consumption caps, and a CLI footgun). All five are fixed. A separate
request-amplification race in the v0.3 verification engine (TP-0, below), discovered while
validating the egress surface, was also fixed.

## True positives

### TP-001 — Malformed-but-200 OSV response crashes the scan  [Severity: MEDIUM]
- **Location:** `gitexpose/supply_chain/osv.py` — `query_osv` (`resp.json()` phase 1; `r.json()` + `_to_vulnerability()` in `_hydrate`).
- **Description:** the network try/except caught only `httpx.HTTPError`. A `200` response with a non-JSON body raises `json.JSONDecodeError` (a `ValueError`), and an unexpected JSON shape makes `_to_vulnerability` raise `AttributeError`/`KeyError`/`TypeError` — none are `httpx.HTTPError`, so they propagated out of `query_osv`, were uncaught in the CLI, and aborted the entire `supply-chain` scan with a traceback.
- **Impact:** availability. The scan dies instead of degrading to the curated list. For a security scanner in CI, an aborted run is a silent coverage gap. Directly violates design spec §9 ("malformed OSV response → skip and continue").
- **Exploit scenario:** non-adversarial and realistic — a captive portal, rate-limit page, or gateway error that returns HTTP 200 with an HTML body; or a partial/truncated JSON response. Adversarially, a TLS MITM (requires a trusted-CA compromise, since httpx validates certs) returning a crafted body.
- **PoC:** `tests/test_osv_client.py::test_query_osv_malformed_querybatch_body_degrades` (200 + `<html>` body) and `::test_query_osv_malformed_vuln_object_skips` (200 + a JSON list where an object is expected). Pre-fix both raise; post-fix both degrade.
- **Fix (shipped):** per-batch and per-vuln parsing is wrapped; failures are logged and skipped, so the scan always degrades to the curated list.
- **References:** CWE-755 (Improper Handling of Exceptional Conditions), CWE-248 (Uncaught Exception).
- **Gates passed:** attacker-influence (incl. non-adversarial), reachable path, real impact (availability), PoC reproduces, devil's-advocate (proxy/gateway 200s make it non-theoretical), spec deviation confirmed.

### TP-002 — OSV-provided vuln ID interpolated into URL path without encoding  [Severity: LOW]
- **Location:** `gitexpose/supply_chain/osv.py` — `_VULN_URL.format(id=vid)`, `vid = v["id"]` from the OSV response.
- **Description:** external data placed into a request path without encoding. A crafted `id` containing `/`, `?`, `#`, or `..` could alter the request path.
- **Impact:** minimal. The host stays `api.osv.dev` and `httpx` defaults to `follow_redirects=False`, so there is **no host pivot and no SSRF** — at most a different path on the already-trusted, HTTPS endpoint. Classed as defense-in-depth.
- **Fix (shipped):** `urllib.parse.quote(vid, safe="")` before interpolation.
- **References:** CWE-116 (Improper Encoding), CWE-20 (Improper Input Validation).

### TP-003 — Unbounded OSV vuln-hydration fan-out  [Severity: LOW-MEDIUM]
- **Location:** `gitexpose/supply_chain/osv.py` — phase 2 hydrated every unique vuln ID; `--osv-max` capped only the *dependencies* queried.
- **Description:** the number of phase-2 GETs is driven by the upstream response (vuln IDs per dependency). A heavily-advisoried package — or a hostile/compromised OSV returning many IDs — produces many outbound requests, throttled by concurrency (8) but with no total cap.
- **Impact:** resource/request amplification from the scanner's egress IP (rate-limit/cost). Same class as the deferred "verify-max total cap" from the v0.3 audit.
- **Exploit scenario:** a querybatch result returning thousands of vuln IDs → thousands of hydration GETs.
- **PoC:** `tests/test_osv_client.py::test_query_osv_caps_hydration_fanout` (6 IDs, `--osv-max 2` → ≤2 GETs).
- **Fix (shipped):** total hydrated IDs are capped to `--osv-max` (sorted for determinism).
- **References:** CWE-770 (Allocation of Resources Without Limits), CWE-799 (Improper Control of Interaction Frequency).

### TP-004 — `parse_all` reads lock files with no size cap  [Severity: LOW]
- **Location:** `gitexpose/supply_chain/lockfiles/base.py` — `parse_all` called `path.read_text()` unbounded.
- **Description:** `LocalFilesystemScanner` caps reads at 1 MB; `parse_all` did not. A crafted multi-GB `package-lock.json`/`poetry.lock` causes memory exhaustion.
- **Impact:** local DoS when scanning an untrusted repository (a realistic mode — e.g., scanning a cloned target or a CI PR). Low; inconsistent with the rest of the scanner.
- **PoC:** `tests/test_lockfile_parse_all.py::test_parse_all_skips_oversized_lockfile` (>1 MB lock file → skipped, empty result).
- **Fix (shipped):** lock files above 1 MB are skipped with a warning, mirroring `LocalFilesystemScanner`.
- **References:** CWE-400 (Uncontrolled Resource Consumption), CWE-770.

### TP-005 — `--osv-max` / `--osv-timeout` lack lower-bound validation  [Severity: LOW]
- **Location:** `gitexpose/cli_advanced.py` — `supply-chain` options.
- **Description:** a negative `--osv-max` slices `deps[:-1]` (silently drops the last dependency); a non-positive `--osv-timeout` yields a degenerate/immediate timeout.
- **Impact:** correctness footgun; no attacker, no security impact. Included for completeness.
- **Fix (shipped):** clamp `--osv-max` to ≥0 and `--osv-timeout` to ≥0.1 in the CLI.
- **References:** CWE-1284 (Improper Validation of Specified Quantity in Input).

### TP-000 — Verification request amplification under concurrency  [Severity: MEDIUM] (verification engine, pre-v0.5)
- **Location:** `gitexpose/verification/engine.py` — `verify_secrets` in-run dedup.
- **Description:** the dedup wrote `seen[secret]` only *after* the `await`, while the membership check ran before any await. Under `asyncio.gather`, identical secrets all missed the cache and each fired an outbound auth request to the provider — defeating the dedup it claimed to provide. (Surfaced as a cross-Python-version-flaky test; root cause is a check-then-act race.)
- **Impact:** K identical findings → K outbound credential-verification requests instead of 1. Request amplification against a third-party provider from the scanner's egress IP — the exact concern the v0.3 audit raised about `--verify` fan-out.
- **PoC:** `tests/test_verification_engine.py::test_dedups_same_secret_within_run` (strengthened to force interleaving; pre-fix `call_count == 3`, post-fix `== 1`).
- **Fix (shipped):** await-once pattern — the first coroutine registers an `asyncio.Event` synchronously (no await between check and register → atomic under cooperative scheduling), computes once, and releases waiters.
- **References:** CWE-362 (Race Condition), CWE-770.

## False positives

None. All five `/attack` candidates and the verification-engine race verified as true positives.

## Negative results (verified safe)

- **No SSRF via file-controlled URLs** — all OSV endpoints are module constants; lock-file-derived data (names, versions, resolved URLs) never reaches URL construction. `resolved_url` is stored for the BOM / future poisoning checks only, never requested in v0.5.
- **No secret egress** — OSV receives only package names + versions; the CycloneDX BOM contains no secrets.
- **CVSS vector parsing** (`cvss.py`) is robust (try/except → `None` → MEDIUM default).
- **yarn.lock regexes** are linear — no ReDoS.
- **JSON/TOML parsing** is stdlib — no code execution.

## Methodology notes

- `fp-check`'s 5-phase verification + 6-gate review was applied **inline** rather than via subagent
  dispatch (subagents were hitting session limits this session). Each true positive is backed by a
  committed regression test that reproduces the pre-fix failure and confirms the post-fix behavior —
  these serve as the positive/negative PoCs the formal phase would generate.
- No high/critical findings, so `second-opinion` was not invoked.
- F-001 is a spec deviation (design §9); anchored to the spec inline rather than via
  `spec-to-code-compliance`.
- Remediations shipped in v0.5.1 — see commits `287eea4` (TP-000), `d3aaae4` (enum-str test),
  `6534e69` (TP-001..TP-005), and `CHANGELOG.md`.
