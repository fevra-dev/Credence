## Reference material reviewed for v0.3 scope

### TruffleHog (industry standard secret scanner)

The headline TruffleHog idea is **Active Verification**: rather than alerting on a string that *looks* like a key, fire a low-footprint authenticated request at the provider's API to confirm the key is live and (optionally) enumerate its permissions.

- Already aligns with GitExpose's `asyncio` engine and low-noise design
- Largest single false-positive reducer for LLM creds in dotfiles
- Upgrades a finding from "exposed pattern" to "confirmed-active critical"
- TruffleHog is AGPL-3.0 Go — we cannot import; we can replicate the per-provider verification HTTP signatures (which are public knowledge: AWS `sts:GetCallerIdentity`, Slack `auth.test`, OpenAI `GET /v1/models`, Anthropic `POST /v1/messages` with 1-token request, GitHub `GET /user`, etc.)
- TruffleHog also has hundreds of Go regex detectors — useful as a structural reference for v0.4+ Tier 4/5 providers

### GitGuardian ggshield

CLI for 350+ secret types with first-class CI integration (pre-commit, GitHub Actions, GitLab CI). Key takeaways:

- **Distribution model**: pre-commit hook + GH Actions workflow are the bread-and-butter integrations that drive adoption. Already on the v0.3 short-list.
- **Honesty about scope**: their docs cleanly separate detectable secret families from upcoming work — informs how we should structure our own COVERAGE.md going forward.
- **API surface**: their `ggshield secret scan` CLI is a good ergonomic reference for our own commands.

### GitGuardian ggcanary

Terraform-managed canary tokens (AWS keys) that page when accessed. Indirectly relevant:

- Nobody is shipping **AI-flavored canary tokens** yet. Concept: planted `sk-ant-canary-…`, `.continue/agents/canary.yaml`, fake `OAI_CONFIG_LIST` with watched creds, fake `litellm_config.yaml`. Hits on these page immediately and indicate an attacker (or an LLM-driven recon agent) is enumerating AI infra.
- This deserves to be its own sister project (GitExpose : ggshield :: GitCanary : ggcanary). NOT v0.3 scope — track for a v0.5+ standalone tool or addon.

### GitGuardian APISecurityBestPractices

Living KB of API security misuse patterns (CWE-style entries with code snippets and fixes).

- Structural inspiration for evolving `docs/COVERAGE.md` into a fuller AI/LLM threat-pattern catalog
- Could ship as `docs/threats/` directory in v0.4 with one .md per attack class (LLM05 supply chain, LLM06 sensitive disclosure, etc.)

### kingbootoshi/cursor-qa

Systematic QA prompting workflow for Cursor IDE. Tangential to GitExpose itself, directly relevant to **how we audit our own work** going forward (see WORKFLOW folder). Use as inspiration for pre-release QA prompts; nothing to port into the repo.

### LinkedIn note: CISA contractor Private-CISA leak

KrebsOnSecurity + GitGuardian reported (May 2026) on a public-but-named-"Private" repo containing 844 MB of CI/CD logs, K8s manifests, Terraform, GitHub workflows, AWS GovCloud admin creds, plaintext passwords. Created Nov 2025, taken down 2026-05-15.

- **Direct validation of the GitExpose thesis** that GitHub is part of the production perimeter
- Worth a 1-paragraph "real-world motivation" callout in README.md (link to Krebs + GitGuardian articles)
- Confirms priorities: secret scanning at org level, short-lived credentials, monitor commits-not-just-runtime
